# AI Engineering Glossary

Practical definitions for terms applied AI engineers encounter in Phases 0-12.
Focuses on terms that are frequently misunderstood or used imprecisely.

| Term | Plain English Definition | Where It Matters In This Curriculum |
|------|--------------------------|--------------------------------------|
| **Token** | The unit an LLM reads and writes. Words, punctuation, and subword fragments are all tokens. "ChatGPT" is one token; "unbelievable" might be three. | Every phase. Tokens determine cost, speed, and context limits. |
| **Tokenization** | The process of converting raw text into tokens before passing it to a model. Different models use different tokenizers, so token counts vary. | Phase 00 foundations; debugging prompt length issues throughout. |
| **Context Window** | The maximum number of tokens a model can process in a single call, including both input and output. Text outside the window is invisible to the model. | Phase 02 (RAG design), Phase 05 (agents), Phase 08 (long-context patterns). |
| **Embedding** | A fixed-length numeric vector that represents the meaning of a piece of text. Texts with similar meanings produce vectors that are close together in vector space. | Phase 02 core concept. Required for all semantic search and RAG pipelines. |
| **Vector** | An ordered list of numbers. In AI engineering, typically refers to an embedding: the numeric representation of text, image, or other content. | Phase 02 throughout. Also Phase 07 (multimodal retrieval). |
| **Cosine Similarity** | A measure of how alike two vectors are based on the angle between them, not their magnitude. Ranges from -1 (opposite) to 1 (identical). | Phase 02 retrieval; used in every vector search ranking step. |
| **Chunking** | Splitting a document into smaller pieces before embedding and indexing. Chunk size and strategy directly affect retrieval quality. | Phase 02 Lesson 04. One of the most impactful tuning decisions in RAG. |
| **RAG (Retrieval-Augmented Generation)** | A pattern where relevant documents are retrieved from a knowledge base and injected into the prompt before generation. Grounds the model's response in external facts. | Phase 02 is entirely RAG. Referenced in every subsequent phase. |
| **Hybrid Search** | Combining dense vector search (semantic similarity) with sparse keyword search (BM25) to improve retrieval coverage and precision. | Phase 02 Lesson 07. Required for production RAG on mixed query types. |
| **BM25** | A classical keyword-based ranking algorithm. Scores documents by term frequency and inverse document frequency. No neural network involved. | Phase 02 Lesson 07. The sparse side of hybrid search. |
| **HNSW (Hierarchical Navigable Small World)** | The graph-based index structure used by most vector databases for approximate nearest neighbor search. Enables fast retrieval at scale. | Phase 02 Lesson 03. Relevant when configuring Qdrant or pgvector. |
| **Reranking** | A second-pass ranking step that takes a candidate set from retrieval and rescores it using a more accurate (but slower) model. | Phase 02 Lesson 11 (advanced RAG). Improves precision after retrieval. |
| **System Prompt** | Instructions sent to the model before the user's message. Defines the model's persona, constraints, output format, and context. | Phase 01 prompt engineering. Foundational to every application built in this curriculum. |
| **Structured Output** | Getting a model to return JSON, XML, or another schema-conforming format rather than free text. Usually enforced via function calling or constrained decoding. | Phase 03 (tool use), Phase 06 (evals), Phase 09 (production APIs). |
| **Function Calling / Tool Use** | A model feature that lets the LLM signal it wants to call an external function, with structured arguments. The host application executes the function and returns the result. | Phase 03 core concept. Required for agents and any real-world integrations. |
| **MCP (Model Context Protocol)** | An open protocol for connecting LLMs to external tools, data sources, and services through a standardized interface. | Phase 04 (MCP integration). Becoming the standard for tool connectivity. |
| **Agent** | An LLM-powered system that can take actions, observe results, and decide what to do next over multiple steps to accomplish a goal. | Phase 05 core concept. Distinct from simple chatbots in architecture and failure modes. |
| **Agentic Loop** | The observe-decide-act cycle an agent runs through repeatedly until it reaches a goal or a stopping condition. | Phase 05 throughout. Understanding this loop is required to debug agent behavior. |
| **Context Engineering** | The discipline of designing what goes into a model's context window: what to include, how to order it, and how to compress it. | Phase 01 and every subsequent phase. Often more impactful than model selection. |
| **Prompt Injection** | An attack where malicious content in the model's input overrides the system prompt or tricks the model into unauthorized actions. | Phase 10 (security). Critical for any agent that processes untrusted input. |
| **OWASP LLM Top 10** | A community-maintained list of the ten most critical security risks for LLM applications, including prompt injection, insecure output handling, and training data poisoning. | Phase 10 security module. The baseline checklist for production LLM security. |
| **Hallucination** | When a model generates text that sounds confident but is factually incorrect or fabricated. A structural property of next-token prediction, not a bug that can be fully patched. | Phase 02 (RAG as mitigation), Phase 06 (evals), Phase 10 (guardrails). |
| **Grounding** | Connecting a model's outputs to verifiable source documents or data. A grounded response can be traced back to specific retrieved content. | Phase 02 Lesson 09. Required for any application where factual accuracy matters. |
| **RAG Triad** | The three dimensions used to evaluate a RAG pipeline: Faithfulness (does the answer match the context?), Answer Relevance (does the answer address the question?), and Context Relevance (did retrieval return the right chunks?). | Phase 02 Lesson 10. The standard framework for RAG evaluation. |
| **LLM-as-Judge** | Using an LLM to evaluate the output of another LLM (or the same one) against a rubric. Enables scalable evaluation without requiring human annotation for every test case. | Phase 06 evals. Central to eval-driven development. |
| **Eval-Driven Development** | Building and iterating AI systems by starting with evaluations, running them on every change, and using metric regressions to guide decisions. | Phase 06. The engineering discipline that separates production AI from demos. |
| **Semantic Caching** | Caching LLM responses based on semantic similarity of the query, not exact string match. A cache hit is triggered when a new query is close enough to a previous one. | Phase 09 (production optimization). Reduces latency and cost at scale. |
| **Prompt Caching** | A feature offered by some LLM providers where a static prefix of the prompt (such as a long system prompt or document) is cached at the API level to reduce cost and latency on repeated calls. | Phase 09 (production optimization). Especially valuable for document-heavy RAG. |
| **Fine-tuning vs RAG** | The choice between adapting a model's weights on domain data (fine-tuning) versus injecting domain knowledge at inference time (RAG). Different tools for different problems. | Phase 02 (RAG) and Phase 11 (fine-tuning). Understanding when each applies prevents costly mistakes. |
| **LoRA / QLoRA** | Parameter-efficient fine-tuning techniques that add a small number of trainable parameters to a frozen model. QLoRA adds quantization to reduce memory requirements further. | Phase 11 (fine-tuning). The standard approach for fine-tuning on consumer hardware or cloud VMs. |
| **Inference** | Running a trained model to generate outputs. Distinct from training. Most applied AI engineering is inference engineering: optimizing how models are called, not how they are trained. | Every phase. The context in which "inference" appears is almost always API calls in this curriculum. |
| **OpenTelemetry GenAI** | An OpenTelemetry semantic convention specification for instrumenting LLM calls, capturing token usage, latency, model parameters, and prompt/completion content in a standardized trace format. | Phase 09 (observability). The standard for adding tracing to production LLM applications. |
| **Forward-Deployed Engineer (FDE)** | An engineer embedded with a customer to build and customize AI systems for their specific workflows. Common role at AI-native companies. | Phase 12 (capstone and career context). Understanding this role shapes what "production-ready" means. |
