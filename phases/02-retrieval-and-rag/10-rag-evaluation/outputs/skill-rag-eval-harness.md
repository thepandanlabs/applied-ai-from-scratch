# Skill: Setting Up a RAG Evaluation Harness

A step-by-step guide for setting up a RAG evaluation harness from scratch: building the eval set, writing judge prompts, running evals, interpreting results, and iterating. Covers the full RAG Triad with production-ready judge prompts.

---

## When to Use This Skill

Use this skill when:
- You've built a RAG system and need to measure quality beyond "it looks OK"
- You're preparing to deploy RAG to production and need a CI gate
- You've changed retrieval strategy, chunk size, or prompts and need to verify no regression
- Someone asks "how do we know the RAG is working?"

---

## Step 1: Do Error Analysis First

Before building any automated metric, **manually read 20–50 traces**. A trace is the full record of one query: retrieved chunks, generated answer, and the final response shown to the user.

For each trace, categorize the failure:

| Failure Type | What You See | Root Cause |
|---|---|---|
| Wrong retrieval | Answer is about a related but different topic | Embeddings not matching query intent |
| Hallucination | Answer adds facts not present in retrieved chunks | Generator using parametric memory |
| Partial answer | Chunks contain the answer but generator missed it | Context window truncation or poor extraction |
| Adjacent answer | Correct topic, wrong specific answer | Ambiguous query + imprecise retrieval |
| Refusal | System declines answerable questions | Over-conservative abstention instruction |
| Out-of-scope leak | System answers when it should abstain | Missing or weak abstention instruction |

**Write down the top 2-3 failure types you see.** These become your priority metrics. Don't build a generic eval suite: build one that measures your actual failure modes.

---

## Step 2: Build Your Eval Set

### Rules for a Good Eval Set

1. **Start with 20-50 real queries** from your application's usage logs or from domain experts. Do not use synthetic queries generated by the same LLM you're evaluating.

2. **Include adversarial examples**: queries that stress-test each failure type you found in Step 1.

3. **Annotate each example** with at minimum:
   - The question
   - What a correct answer looks like (or which chunks should be retrieved)
   - Whether the question is answerable from your corpus

### Minimal eval set schema

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class EvalExample:
    question: str
    context: list[str]    # retrieved chunk texts (from your actual retriever)
    answer: str           # generated answer (from your actual generator)
    expected_answer: Optional[str] = None
    is_answerable: bool = True
    failure_category: Optional[str] = None  # from your error analysis
```

### Getting the context and answer

Run your RAG pipeline on each query and save the full trace:

```python
def build_eval_example(query: str, rag_pipeline) -> EvalExample:
    retrieved_chunks, answer = rag_pipeline.run_with_trace(query)
    return EvalExample(
        question=query,
        context=[c.text for c in retrieved_chunks],
        answer=answer,
    )
```

## Step 3: Judge Prompts for All Three RAG Triad Components

### Faithfulness Judge

**Purpose:** Is every claim in the answer supported by the context?

```
System:
You are an objective evaluator. Your task: assess whether a given ANSWER is
faithful to (directly supported by) the provided CONTEXT.

Steps:
1. Identify each distinct factual claim in the ANSWER.
2. For each claim, determine if the CONTEXT explicitly entails it.
   Do not infer. Require direct textual support.
3. Return JSON:
{
  "claims": [
    {"claim": "<text>", "supported": true/false, "reason": "<why>"}
  ],
  "faithfulness_score": <fraction supported, 0.0-1.0>
}

Rules:
- Citation markers like [1] in the answer are decorative; ignore them.
- If the answer contains zero factual claims, return faithfulness_score: 1.0.
- "Supported" means the CONTEXT explicitly says this. Not "consistent with": explicitly says it.

User:
QUESTION: {question}
CONTEXT: {context}
ANSWER: {answer}
```

### Answer Relevance Judge

**Purpose:** Does the answer address the user's actual question?

```
System:
You are an objective evaluator. Rate how well the ANSWER addresses the QUESTION.

Scoring:
- 1.0: Directly and completely answers the question
- 0.8: Addresses the main question, misses secondary aspects
- 0.5: Topically related but doesn't answer the specific question
- 0.2: Related topic, different specific answer
- 0.0: Off-topic, complete refusal, or entirely irrelevant

Return JSON:
{
  "answer_relevance_score": <float 0.0-1.0>,
  "reasoning": "<1-2 sentences>"
}

User:
QUESTION: {question}
ANSWER: {answer}
```

### Context Relevance Judge

**Purpose:** Are the retrieved chunks actually useful for answering the question?

```
System:
You are an objective evaluator. Assess whether each retrieved CHUNK helps
answer the QUESTION.

A chunk is relevant if it directly contributes to answering the question.
A chunk is NOT relevant if it only shares keywords without helping construct the answer.

Return JSON:
{
  "chunk_scores": [
    {
      "chunk_index": <int>,
      "relevant": true/false,
      "relevance_score": <float 0.0-1.0>,
      "reasoning": "<1 sentence>"
    }
  ],
  "context_relevance_score": <mean score, 0.0-1.0>
}

User:
QUESTION: {question}
CHUNKS: {chunks}
```

## Step 4: Run the Harness

```python
import os
from openai import OpenAI

evaluator = RagEvaluator(model="gpt-4o-mini")
results = evaluator.evaluate_batch(eval_examples)

print(f"Faithfulness:     {results['mean_faithfulness']:.3f}")
print(f"Answer Relevance: {results['mean_answer_relevance']:.3f}")
print(f"Context Relevance:{results['mean_context_relevance']:.3f}")
```

Target baselines for a production-ready RAG system:

| Metric | Minimum | Target |
|---|---|---|
| Faithfulness | 0.80 | 0.90+ |
| Answer Relevance | 0.75 | 0.85+ |
| Context Relevance | 0.70 | 0.80+ |

---

## Step 5: Calibrate the Judge

Before trusting automated scores at scale, validate the judge against human labels.

```python
# Collect human scores on 20 examples
human_faithfulness = [1.0, 0.8, 0.5, 1.0, 0.0, ...]  # hand-labeled

# Run judge on same examples
judge_faithfulness = [evaluator.evaluate(ex)["faithfulness"] for ex in examples[:20]]

calibration = calibrate_judge(human_faithfulness, judge_faithfulness)
print(f"Agreement: {calibration['agreement_rate']:.1%}")
# Target: >= 85%
```

If agreement < 85%:
1. Add 3-5 few-shot examples to the judge system prompt (one correct, one over-strict, one over-lenient)
2. Try a stronger judge model (gpt-4o instead of gpt-4o-mini)
3. Decompose the metric more finely (per-sentence faithfulness instead of per-answer)

## Step 6: Interpret Results Using the Diagnostic Matrix

| Faithfulness | Answer Relevance | Context Relevance | Action |
|---|---|---|---|
| Low | Any | Low | Fix retrieval first: better chunking, embeddings, or reranking |
| Low | High | High | Generator hallucinating: tighten grounding prompts, lower temperature |
| High | Low | High | Generator answering adjacent question: check query transformation |
| High | High | Low | Getting lucky: retrieval will fail on harder queries eventually |
| All high | | | Working. Keep monitoring. |

**Priority order for fixing:**
1. Context Relevance < 0.7 → fix retrieval
2. Faithfulness < 0.8 → fix generation prompts
3. Answer Relevance < 0.75 → fix query understanding or reranking

## Step 7: Integrate into CI

```yaml
# .github/workflows/rag-eval.yml
name: RAG Evaluation Gate

on:
  pull_request:
    paths:
      - 'prompts/**'
      - 'retrieval/**'
      - 'chunking/**'

jobs:
  rag-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run RAG Eval Harness
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: python evals/run_rag_triad.py --fail-on-regression
```

```python
# evals/run_rag_triad.py
import sys
from rag_evaluator import RagEvaluator, EVAL_SET

evaluator = RagEvaluator()
results = evaluator.evaluate_batch(EVAL_SET)
check = evaluator.check_thresholds(results)

if not check["passed"]:
    print("RAG eval FAILED:")
    for failure in check["failures"]:
        print(f"  - {failure}")
    sys.exit(1)

print(f"RAG eval PASSED: {check['scores']}")
```

## Step 8: Iterate

The eval loop is:

```
1. Run eval harness → get scores
2. Look at per-example failures (not just aggregates)
3. Categorize failures by type (from Step 1 taxonomy)
4. Pick the highest-frequency failure type
5. Fix it (change chunking / retrieval / prompt)
6. Re-run eval harness
7. Verify the target metric improved without regressing others
8. Repeat
```

Never fix a failure mode without re-running the full harness. Prompt changes that fix faithfulness often hurt answer relevance. The triad catches cross-metric regressions.

---

## Common Mistakes

| Mistake | What Goes Wrong | Fix |
|---|---|---|
| Using synthetic eval queries from the same LLM as the generator | Distribution collapse: the eval set mirrors the generator's strengths | Use real user queries or domain expert queries |
| Trusting scores without calibration | Judge may be systematically wrong for your domain | Always calibrate on 20 human-labeled examples |
| Only looking at aggregate scores | A mean of 0.8 can hide 5 examples scoring 0.0 | Always review per-example failures |
| Building the eval set before doing error analysis | Metrics don't cover your actual failure modes | Error analysis first, always |
| Fixing one metric without re-running others | Prompt changes create cross-metric regressions | Always run the full triad after any change |
