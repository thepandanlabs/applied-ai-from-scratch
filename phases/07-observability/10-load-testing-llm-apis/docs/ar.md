# اختبار الحمل (Load Testing) لواجهات الـ LLM البرمجية

> أجرِ الاختبار قبل أن يجريه الترافيك. اختبارات الحمل في الإنتاج اسمها حوادث (incidents).

**النوع:** بناء
**اللغات:** Python
**المتطلبات:** المرحلة 07 الدروس 01-09 (الـ observability، والتوجيه)، المرحلة 06 (أساسيات الإطلاق)
**الوقت:** ~45 دقيقة
**أهداف التعلّم:**
- شرح لماذا تحتاج أدوات اختبار حمل HTTP المعيارية إلى تكييف لتناسب واجهات الـ LLM
- تنفيذ مشغّل اختبار حمل قائم على asyncio يجمع TTFT، وإجمالي زمن الاستجابة، ومعدّل الخطأ، والتكلفة
- تفسير تقرير ملخّص اختبار الحمل لتحديد عنق الزجاجة
- استخدام locust لسيناريو ramp-up واقعي
- فهم بروتوكول أمان التكلفة عند اختبار الحمل مقابل واجهات API حقيقية

---

## المشكلة

يُطلق فريقك ميزة ذكاء اصطناعي جديدة. يتصاعد الترافيك بعد الإطلاق. عند 50 مستخدمًا متزامنًا، تقفز أزمنة الاستجابة من ثانيتين إلى 18 ثانية، وتبدأ بعض الطلبات بإرجاع أخطاء 429. لم يسبق لمهندس المناوبة (on-call) أن رأى نمط الفشل هذا.

ستخبرك أدوات اختبار حمل HTTP المعيارية مثل k6 أو wrk بعدد الطلبات/الثانية ومعدّل الخطأ. لكن لواجهات الـ LLM ثلاث خصائص تجعل اختبار الحمل المعياري مضلّلًا:

أولًا، أزمنة الاستجابة متغيّرة بمقدار 10-100 ضعف بحسب طول الإخراج. استجابة من 50 توكن تستغرق ثانية واحدة؛ واستجابة من 2,000 توكن تستغرق 20 ثانية. معاملة هذين على أنهما "الطلب" نفسه في قياس p99 يُخفي توزيع زمن الاستجابة الفعلي.

ثانيًا، تعني الاستجابات المتدفقة (streaming) أن زمن الوصول إلى أول توكن (TTFT) مقياس مستقل عن إجمالي زمن الاستجابة. يدرك المستخدمون الـ TTFT على أنه سرعة الاستجابة. تجربة TTFT يبلغ 500ms مع إجمالي مدته 15 ثانية مختلفة جدًا عن تجربة TTFT يبلغ 5 ثوانٍ مع إجمالي مدته 8 ثوانٍ.

ثالثًا، تتفاعل حدود المعدّل (rate limits) مع التزامن بطرق غير بديهية. واجهة Anthropic البرمجية تفرض حدودًا على التوكنات في الدقيقة، لا على الطلبات في الدقيقة فقط. دفقة من طلبات السياق الطويل قد تصطدم بحد التوكنات قبل حد الطلبات.

تحتاج إلى اختبار حمل يقيس الأشياء الصحيحة دون أن يفلسك في العملية.

## المفهوم

### توقيت الطلبات المتزامنة

```
Time (seconds) -->

req-01  |====QUEUE===|=TTFT=|=========STREAM=========|  done at t=12
req-02  |====QUEUE===|=TTFT=|======STREAM======|        done at t=9
req-03  |====QUEUE===|==TTFT==|============STREAM============|  done at t=16
req-04            |=QUEUE=|=TTFT=|====STREAM====|            done at t=11
req-05            |=QUEUE=|==TTFT==|=========STREAM=========| done at t=15

         ^ concurrency starts here
                     ^ first token for each request (TTFT = queue + model startup)
                                   ^ streaming output (cost of output tokens)

Rate limit window (1 minute):
[####################] 40% token budget used by req-01..05
[########] 20% more from req-06..10
[##############################] Rate limit hit at req-17 -> 429
```

ملاحظات أساسية:
- الـ TTFT هو ما يراه المستخدم أولًا. يشمل زمن الشبكة، والوقوف في طابور الـ API، وتهيئة النموذج.
- إجمالي زمن الاستجابة = TTFT + زمن الـ streaming. زمن الـ streaming يعتمد على طول الإخراج.
- اصطدامات حد المعدّل غير مرئية حتى تحدث. أجرِ الاختبار على مراحل: 10 طلبات، ثم 25، ثم 50.

### ماذا تقيس

| المقياس | لماذا يهم | كيف تجمعه |
|--------|---------------|----------------|
| TTFT p50/p95 | سرعة الاستجابة كما يدركها المستخدم | الطابع الزمني عند أول chunk |
| Total latency p50/p95 | زمن الانتظار من الطرف إلى الطرف | الطابع الزمني عند آخر chunk |
| Error rate | حدود المعدّل وأخطاء الخادم | عُدّ الاستجابات غير 200 |
| Requests/sec throughput | مستوى التزامن المستدام | إجمالي الطلبات / مدة الاختبار |
| Estimated cost | أثر هذا الترافيك على الميزانية | مجموع توكنات الإدخال+الإخراج × السعر |

## البناء

ثبّت المتطلبات:

```bash
pip install anthropic httpx
```

يطلق صنف `LLMLoadTester` عددًا N من الطلبات المتزامنة، ويجمع TTFT وإجمالي زمن الاستجابة لكل منها، ويطبع تقريرًا مُلخَّصًا. يستهدف خادمًا وهميًا (mock) افتراضيًا حتى تتمكن من التحقق من بنية الاختبار قبل إنفاق المال.

```python
import asyncio
from load_tester import LLMLoadTester, LoadTestConfig

config = LoadTestConfig(
    concurrency=10,
    total_requests=50,
    prompt="Explain the tradeoffs between synchronous and asynchronous Python in 100 words.",
    max_tokens=150,
    use_mock=True,   # set False to hit the real Anthropic API
)

async def run():
    tester = LLMLoadTester(config)
    report = await tester.run()
    report.print_summary()

asyncio.run(run())
```

المخرجات المتوقعة (وضع الـ mock):

> **اختبار من الواقع:** لماذا نختبر مقابل خادم وهمي (mock) أولًا؟ لأن بنية اختبار الحمل نفسها بها أخطاء عند أول كتابة. إذا شغّلت 500 طلب مقابل الـ API الحقيقي لتكتشف أن منطق التزامن لديك به deadlock، فأنت قد أنفقت 5 دولارات للعثور على خطأ كان بإمكانك العثور عليه مجانًا. الـ mock يتحقق من كود القياس لديك، ومن معالجة الأخطاء، ومن نموذج التزامن. لا تشغّل مقابل الـ API الحقيقي إلا بعد أن يُنتج تشغيل الـ mock نتائج نظيفة.

شغّل في وضع الـ mock:

```bash
python code/main.py --mock --concurrency 10 --requests 50
```

ثم أجرِ اختبارًا حقيقيًا صغيرًا (يتحقّق من أرقام زمن الاستجابة الفعلية):

```bash
ANTHROPIC_API_KEY=your_key python code/main.py --concurrency 5 --requests 20
```

## الاستخدام

يوفّر Locust واجهة ويب، ومنحنيات ramp-up، وتوليد حمل موزّع عندما تتجاوز إمكانات المشغّل غير المتزامن (async runner):

```python
from locust import HttpUser, task, between
import json

class LLMUser(HttpUser):
    wait_time = between(1, 3)

    @task
    def query(self):
        payload = {
            "model": "claude-3-5-haiku-20241022",
            "messages": [{"role": "user", "content": "Summarize AI in one sentence."}],
            "max_tokens": 100,
        }
        headers = {"x-api-key": self.environment.host}
        with self.client.post(
            "https://api.anthropic.com/v1/messages",
            json=payload,
            headers={"anthropic-version": "2023-06-01", **headers},
            catch_response=True,
            name="llm_query",
        ) as resp:
            if resp.status_code == 429:
                resp.failure("rate_limited")
            elif resp.status_code != 200:
                resp.failure(f"http_{resp.status_code}")
```

شغّل بـ: `locust -f locustfile.py --users 20 --spawn-rate 2 --run-time 60s --headless`

> **نقلة في المنظور:** الـ ramp-up في Locust (المعامل spawn-rate) هو أهم معامل في اختبارات حمل الـ LLM، لا التزامن الأقصى (peak concurrency). بدء 20 مستخدمًا في آنٍ واحد يصطدم بالـ API قبل أن يُسخّن أي مجمّعات اتصال (connection pools) أو يخصّص أي سياق مسبقًا. الترقّي بمعدّل مستخدمَين في الثانية يمنحك صورة واقعية عمّا يحدث مع نمو الترافيك، ويتيح لك تحديد مستوى التزامن الذي تبدأ عنده حدود المعدّل بالظهور، بدلًا من المستوى الذي يكون فيه كل شيء معطّلًا أصلًا.

## التقييم

**ابنِ خط أساس قبل أن تحتاجه:** أجرِ اختبار الحمل عند تزامن 10 و25 و50 قبل إطلاقك. سجّل p95 لـ TTFT وإجمالي زمن الاستجابة عند كل مستوى. هذا هو خط أساس الأداء لديك.

**اكتشاف حد المعدّل:** ارفع التزامن حتى ترى أول 429. مستوى التزامن الواقع تمامًا أسفل تلك العتبة هو حدّ التشغيل الآمن لديك. اضبط مُحفّزات التوسّع التلقائي (auto-scaling) عند 70% من ذلك الحد.

**التكلفة لكل تشغيل اختبار:** ينبغي للاختبار الصغير على الـ API الحقيقي (20 طلبًا عند تزامن 5) أن يكلّف أقل من 0.01 دولار مع Haiku. إذا كلّف أكثر، فتحقّق من أن `max_tokens` مضبوط بتحفّظ. لا تشغّل الاختبار الكبير (تزامن 50، 200 طلب) إلا عند التحقق لإطلاق عالي الترافيك، وخصّص له ميزانية صراحةً.

**عتبات التنبيه:** اضبط تنبيهًا إذا تجاوز p95 لـ TTFT ثلاثة أضعاف خط أساسك. قفزة زمن استجابة مفاجئة عند مستوى الترافيك نفسه تعني عادةً أن مزوّد الـ API يواجه حادثًا، لا أن كودك تغيّر.
